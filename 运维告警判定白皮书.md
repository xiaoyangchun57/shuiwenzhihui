## 1. 运维职责边界

### 1.1 核心原则

水利运维智慧运营平台的告警检测范围限于 **"设备-通信-数据链路"** 这一层，不越界到 **"业务指标超限"** 层。区分依据：

| 维度 | 运维告警（本平台履职） | 业务告警（外部门履职） |
|------|----------------------|----------------------|
| 关注对象 | 设备是否正常工作、数据是否正常上报 | 水文/气象/水质指标是否超过安全阈值 |
| 典型问题 | 设备离线、传感器故障、通信中断、电源异常 | 水位超警、降雨超阈值、水质超标 |
| 判断依据 | 设备状态、数据行为模式（缺失/冻结/突变） | 行业规范规定的阈值（如警戒水位） |
| 主管部门 | 运维中心/工管科 | 水文局/气象局/环保局 |
| SL标准对应 | SL 551 第5章：监测设施维护 | SL 258 第6章：大坝安全评价 |

### 1.2 该管清单（运维核心职责）

| 序号 | 告警类型 | 说明 | 行业依据 |
|------|---------|------|---------|
| 1 | **设备离线/失联** | 通信中断、电源掉电、模块死机，连续N个周期无心跳 | SL 551-2012 第5.1.3条：监测设施应定期检查，发现损坏及时修复 |
| 2 | **数据停报** | 超过N分钟未收到任何数据包（不区分指标） | SL 551-2012 第5.3.1条：数据采集系统故障应及时排除 |
| 3 | **传感器数值冻结** | 连续多条记录值完全一致（传感器卡死） | SL 551-2012 第5.3.4条：监测数据出现异常应检查仪器工作状态 |
| 4 | **传感器数值突变** | 数值跳变超出物理可能范围或统计异常 | SL 601-2013 第6.0.5条：监测数据应进行合理性检查 |
| 5 | **电压异常** | 电池电压低于11.8V（铅酸）/低于3.3V（锂电池）/过压 | SL 551-2012 第5.1.5条：电源系统应定期维护 |
| 6 | **站房环境异常** | 站房温度/湿度超限（如机柜温度>45°C） | SL 725-2016 第4.2条：作业环境应符合安全要求 |
| 7 | **通信信号弱** | 4G/NB-IoT信号强度低于-110dBm | 运维实践惯例 |
| 8 | **到报率不足** | 当日到报率低于阈值（如<90%） | 水利部数据质量管理办法 |

### 1.3 不该管清单（明确排除）

| 序号 | 指标 | 归属部门 | 排除理由 | 例外情况 |
|------|------|---------|---------|---------|
| 1 | 降雨量超过阈值（如>50mm/h） | 气象局/水文局 | 属于气象预警，平台不应重复发布 | 若本站与邻近3站数据一致但本站突变为0→设备故障，属运维 |
| 2 | 水位超过警戒线/保证水位 | 水文局 | 属于水文预警（SL 258-2017 第6.3条） | 若水位值跳变3m+且无法合理解释→传感器故障，属运维 |
| 3 | 河流流量预测/洪水预报 | 水文局 | 属于水文预报业务 | 不涉及 |
| 4 | 水质指标超标（浊度/pH/余氯等） | 环保部门/水厂 | 属于水质管理业务 | 若多个指标同时归零→设备故障，属运维 |
| 5 | 土壤墒情干旱等级 | 农业/防汛部门 | 属于旱情评估 | 传感器值持续不变→冻结检测 |

### 1.4 边界案例判定规则

| 场景 | 判定 | 理由 |
|------|------|------|
| 降雨量为0持续48h | **不是告警**（雨量站无雨时零值正常） | 自然现象，见第3.2节 |
| 水位为0持续12h | **可能是告警**（水位站干涸需要确认） | 干河床可能，但需与邻近站比对确认非传感器故障 |
| 降雨量瞬间从0跳变到200mm | **不是降雨告警**，**但可能是运维告警** | 物理不可能，疑似传感器故障→触发 data_spike 检测 |
| 某站气温50°C但邻近站30°C | **可能是运维告警** | 传感器故障可能性大，现场核实 |
| 系统检测到所有指标同时归零 | **肯定是运维告警** | 极可能是设备掉电/主控板故障 |

### 2.1 越界问题（不该管的管了）

| 编号 | 问题位置 | 当前行为 | 问题分析 | 修正建议 |
|------|---------|---------|---------|---------|
| **C1** | detect_site_anomalies L1057-1063 | 冻结检测对所有指标一视同仁，包括 `precipitation` 在内的冻结检测被 `EXCLUDE_SPIKE` 排除 | **正确排除**了 `precipitation` 的冻结检测。但 `EXCLUDE_SPIKE` 名称语义有误导（它实际排除了冻结+突变两种检测），建议改名 | 见3.2节 |
| **C2** | detect_site_anomalies L1066-1089 | 突变检测对 `precipitation` 等已正确排除。但 `min_abs_change` 表未覆盖所有指标 | `velocity` 无阈值、`temperature` 阈值5°C偏大（气温日较差一般不超过15°C但单次涨5°C也合理）、`evaporation` 阈值3偏大 | 补全所有指标的 `min_abs_change` |

### 2.2 逻辑不完善问题

| 编号 | 问题位置 | 当前行为 | 问题分析 | 修正建议 |
|------|---------|---------|---------|---------|
| **L1** | detect_site_anomalies L1058 | 冻结检测：`len(set(...)) == 1` 要求**连续6条完全相同** | 过于严格。实际传感器可能因量化误差在±1 LSB抖动，应允许极小浮动（如 ±0.01% 范围） | 改为`max - min < epsilon`，不同指标`epsilon`不同 |
| **L2** | detect_site_anomalies L1092-1101 | 缺失检测：只比较最新两条时间戳的差值 | 如果数据采集周期是5分钟，但第3条开始停报，第6条恢复——这中间只有2条记录，但缺失时长远大于阈值。当前逻辑只能检测"最新一条未更新"，无法检测"中间缺失" | 应改为比较"最新记录时间与当前时间"的差值 |
| **L3** | detect_site_anomalies L1074-1078 | z_score 计算使用6条样本，样本量太小，标准差估计不稳定 | 小样本下1个离群值就会严重拉偏标准差，导致z_score失效 | 建议使用滑动窗口（如30条历史数据）或固定基线 |
| **L4** | 缺失检测 L1097 | 阈值固定25分钟 | 不同站点类型采集周期不同（降雨5分钟、水位15分钟），固定阈值不适用 | 改为按采集周期动态计算 |
| **L5** | create_alert_internal L1281 | `A_LEVEL_METRICS` 只包含 `data_gap` 和 `device_status` | `data_spike`（红级）和 `data_freeze`（多次触发后）也应该能自动转工单 | 见第5节 |
| **L6** | 整体 | `data_spike` 的 `min_abs_change` 字典缺少 `velocity` 等指标，退化到 `abs(mean)*0.25` | 对于接近零的均值，退化值可能极小（如均值为0.1m/s时，min_abs=0.025m/s），导致误报 | 补全所有指标的物理合理阈值 |

### 2.3 阈值合理性评估

| 指标 | 当前阈值 | 行业参考值 | 评估 | 建议 |
|------|---------|-----------|------|------|
| 水位突降min_abs | 0.5m | 水位传感器精度0.01m，正常变化<0.1m/10min | 合理 | 保持 |
| 流量突降min_abs | 500m³/s | 依河段不同差异大 | 偏大（中小河流最大流量不到500） | 改为相对值`abs(mean)*1.5`，下限`min(mean*1.5, 100)` |
| 土壤含水量突降min_abs | 8% | 日变化一般<2% | 合理但偏保守 | 可降至5%以提高灵敏度 |
| 气温突降min_abs | 5°C | 10分钟内变化>3°C已属异常 | 偏大 | 改为3°C |
| 蒸发量突降min_abs | 3mm | 日蒸发量一般2-8mm | 对于10分钟粒度偏大 | 改为1mm |
| 地下水位突降min_abs | 3m | 10分钟内变化>0.5m已异常 | 严重偏大 | 改为0.5m |
| 数据缺失阈值 | 25min | 依采集周期而定（常规5min×3个周期=15min） | 偏松 | 改为`采集周期 × 3` |
| 冻结检测连续条数 | 6条 | — | 合理 | 保持，但放宽精度至±0.01% |

## 3. 5种运维告警判定规则

### 3.1 data_gap — 数据缺失

| 项目 | 内容 |
|------|------|
| **含义** | 站点在某指标上超过规定时间未上报数据 |
| **触发条件** | 当前时间 - 最新记录时间 > `采集周期 × 3`（其中采集周期依站点类型确定） |
| **采集周期表** | 雨量站=5min，水文站=5min，水位站=15min，地下水位站=30min，墒情站=60min，蒸发站=60min，站院环境=5min |
| **级别判定** | **Yellow**：超时3~6个周期；**Orange**：超时6~12个周期；**Red**：超时>12个周期或超时>2h |
| **自动/复核** | **自动转工单**（A级告警），因设备明显失联 |
| **适用的指标** | 所有指标 |
| **不适用的指标** | 无（当站点所有指标都停报时，只产生一条告警，避免风暴） |
| **防误报策略** | 站点状态为"计划停机维护"时不触发；判断整站停报（所有指标同时缺失）和单指标停报 |
| **当前代码状态** | ✅ 已有检测，但使用固定25min阈值（L1097），需要修正为动态周期 |

**修正细则：**

```python
# 采集周期定义（分钟）
COLLECT_INTERVAL = {
    'rainfall': 5,
    'hydrology': 5,
    'water_level': 15,
    'groundwater': 30,
    'soil_moisture': 60,
    'evaporation': 60,
    'station_yard': 5,
}
gap_minutes = (now - latest_time).total_seconds() / 60
threshold = COLLECT_INTERVAL.get(site_type, 15) * 3
if gap_minutes > threshold:
    # yellow: 3~6周期; orange: 6~12周期; red: >12周期
    ...
```

### 3.2 data_freeze — 数据冻结

| 项目 | 内容 |
|------|------|
| **含义** | 传感器数值连续多条记录几乎不变，疑似传感器卡死/堵头/线缆断 |
| **触发条件** | 连续N条记录的最大值与最小值之差 < `epsilon`（epsilon按指标设定） |
| **N值** | 冻结窗口 = 6条（约30min~6h，取决于采集周期） |
| **epsilon表** | 水位=0.01m，流量=0.1m³/s，地下水位=0.01m，土壤含水量=0.1%，气温=0.1°C，蒸发量=0.1mm，流速=0.01m/s，水质=0.01 |
| **级别判定** | **Yellow**：首次检测到冻结（<3次触发）；**Orange**：同指标第3次触发冻结；**Red**：同一站点同时3个及以上指标冻结 |
| **自动/复核** | **人工复核**（B级告警），因冻结可能是真故障也可能是监测对象确实静止（如枯水期地下水） |
| **适用的指标** | water_level, flow, velocity, groundwater_level, soil_moisture, soil_temperature, evaporation, temperature, water_quality |
| **明确不适用的指标** | **precipitation**（无雨时zero是正常现象）、**cumulative_rainfall**（累积值在无雨区间自然不变）、**noise**（夜间可能稳定在低值）、**wind_speed**（静风期可能为0） |
| **防误报策略** | ①降水为0持续不变不告警（已在排除清单）；②枯水期水位为0持续不变不告警（需结合站点上下游历史判断）；③所有传感器同时冻结→可能为采集器死机，升级为Red |
| **当前代码状态** | ✅ 已有检测（L1057-1063），但：①目前使用 `len(set())==1` 过于严格；②未给出按指标的epsilon；③排除列表 `EXCLUDE_SPIKE` 命名不当（实际也是冻结排除列表） |

**修正细则：**

```python
FREEZE_EPSILON = {
    'water_level': 0.01, 'flow': 0.1, 'velocity': 0.01,
    'groundwater_level': 0.01, 'soil_moisture': 0.1,
    'soil_temperature': 0.1, 'evaporation': 0.1,
    'temperature': 0.1, 'water_quality': 0.01,
}
EXCLUDE_FREEZE = {'precipitation', 'cumulative_rainfall', 'noise', 'wind_speed'}

if metric not in EXCLUDE_FREEZE and len(values) >= 6:
    eps = FREEZE_EPSILON.get(metric, 0.01)
    if max(values[:6]) - min(values[:6]) < eps:
        # → 冻结告警
```

### 3.3 data_spike — 数据异常突变

| 项目 | 内容 |
|------|------|
| **含义** | 传感器数值出现物理不可能或统计极显著的跳变 |
| **触发条件** | 满足以下**全部**条件：①当前值偏离前5条均值的百分比 > 15%；②z_score > 6；③绝对变化值 > `min_abs` |
| **min_abs表** | 水位=0.5m，流量=`max(mean*1.5, 100)`m³/s，流速=0.5m/s，地下水=0.5m，土壤含水量=5%，气温=3°C，蒸发量=1mm，水质=0.5 |
| **z_score计算** | 使用**滑动窗口**（前30条历史数据）计算均值和标准差，而非仅前5条 |
| **级别判定** | **Orange**：z_score > 6且变化>15%；**Red**：z_score > 10 或 同一站点30分钟内多个指标同时突变 |
| **自动/复核** | **人工复核**（B级告警），需现场确认是真突变还是传感器故障 |
| **适用的指标** | water_level, flow, velocity, groundwater_level, water_quality, temperature |
| **明确不适用的指标** | **precipitation**（降雨本身就是突变的）、**cumulative_rainfall**（自然递增）、**wind_speed**（阵风自然突变）、**noise**（瞬时噪声正常）、**soil_moisture**（变化缓慢，需降低灵敏度）、**evaporation**（日变化有规律） |
| **防误报策略** | ①均值接近零时(mean<0.001)跳过；②所有指标同时突变→可能为采集器故障，升级为Red；③邻近站比对：若邻近3站趋势一致则本站非故障；④突变后回归正常→可能为干扰信号，降级 |
| **当前代码状态** | ✅ 已有检测（L1066-1089），问题：min_abs表不全、z_score基于小样本、缺少邻近站比对 |

**修正细则：**

```python
SPIKE_MIN_ABS = {
    'water_level': 0.5, 'flow': None,  # 动态计算
    'velocity': 0.5, 'groundwater_level': 0.5,
    'soil_moisture': 5.0, 'temperature': 3.0,
    'evaporation': 1.0, 'water_quality': 0.5,
}

def _calc_min_abs(metric, mean):
    if metric == 'flow':
        return max(mean * 1.5, 100)
    return SPIKE_MIN_ABS.get(metric, abs(mean) * 0.25)
```

**土壤含水量特殊处理：** 墒情站 data_spike 灵敏度降低50%（阈值翻倍），因土壤含水量自然变化缓慢。

### 3.4 device_status — 设备异常

| 项目 | 内容 |
|------|------|
| **含义** | 设备离线、电压异常、通信信号弱、站房环境异常 |
| **触发条件** | 多个子类型：①**设备离线**：设备shadow表status=offline；②**电压低**：`voltage < 11.8V`（铅酸）或 `<3.3V`（锂电）；③**电压过压**：`voltage > 15V`（铅酸）；④**信号弱**：`RSSI < -110dBm`；⑤**站房高温**：`temperature > 45°C` 或 `humidity > 85%` |
| **级别判定** | **Yellow**：离线<1h 或 电压低(>12V) 或 信号弱；**Orange**：离线1~4h 或 电压<12V 或 站房超限；**Red**：离线>4h 或 电压<11V 或 同时多个设备离线 |
| **自动/复核** | **自动转工单**（A级告警），因设备离线/电源故障需要立刻派人处理 |
| **适用的站点** | 所有站点类型 |
| **不适用的指标** | 无 |
| **防误报策略** | 已知计划内关机不告警（通过维护计划表校验）；电压恢复后自动解除告警（恢复检测） |
| **当前代码状态** | ✅ 离线告警在 `generate_sensor_data()` 中（L1190-1198）已有。但缺少电压检测、信号检测、站房环境检测。需补全 |

### 3.5 arrival_rate — 到报率不足

| 项目 | 内容 |
|------|------|
| **含义** | 站在指定时间段内，实际收到的数据包数量未达到预期 |
| **触发条件** | 当日（自然日00:00-23:59）到报率 = `实际收到数 / 预期收到数 × 100%` < 阈值 |
| **阈值** | **Yellow**：到报率 80%~90%；**Orange**：60%~80%；**Red**：<60% |
| **预期数计算** | 24h × 60min / 采集周期（如雨量站=5min，预期=288条） |
| **级别判定** | Yellow: 80%-90%; Orange: 60%-80%; Red: <60% |
| **自动/复核** | **自动转工单**（A级告警），到报率不足说明通信链路或设备有问题 |
| **适用的站点** | 所有站点类型（按各自的采集周期计算） |
| **防误报策略** | ①新投运站点（投运<3天）不触发；②计划内停机日不统计；③连续3天到报率不足才生成Red级别 |
| **当前代码状态** | ❌ 当前只有到报率**统计功能**（`data_arrival`表+L2656-L2697），**没有到报率不足的告警生成逻辑**。这是缺失的功能，需新增 |

## 4. 站点类型与告警映射矩阵

### 4.1 完整映射矩阵

| 站点类型 | 适用告警类型 | 不适用的告警 | 特殊说明 |
|---------|-------------|-------------|---------|
| **hydrology**（水文站） | data_gap, data_freeze, data_spike, device_status, arrival_rate | 无 | 监测水位+流量+流速。data_spike 对水位和流量都启用，min_abs 分别设定 |
| **water_level**（水位站） | data_gap, data_freeze, data_spike, device_status, arrival_rate | 无 | 仅监测水位。冻结检测 epsilon=0.01m |
| **rainfall**（雨量站） | data_gap, device_status, arrival_rate | **data_freeze**（零值正常）、**data_spike**（降雨自然突变） | 核心规则：雨量站不做冻结检测（无雨时降水=0是正常现象）。只做data_spike的**同区域一致性比对**——如邻近3站降水均为0但本站突然报50mm，则可能是传感器误触发（翻斗卡住等），应标记为可疑。但确认属运维范畴 |
| **groundwater**（地下水位站） | data_gap, data_freeze, data_spike, device_status, arrival_rate | 无 | 地下水变化缓慢，spike检测的min_abs=0.5m（而非水位站的0.5m，实际可更保守） |
| **soil_moisture**（墒情站） | data_gap, data_freeze, device_status, arrival_rate | **data_spike**（墒情变化极其缓慢） | 土壤含水量日变化一般<2%，突变检测灵敏度应降低（min_abs=5%或关闭）。但冻结检测仍然启用 |
| **evaporation**（蒸发站） | data_gap, data_freeze, device_status, arrival_rate | **data_spike**（蒸发量日变化有固定规律） | 蒸发量受日照/气温影响，日内变化有规律，不建议突变检测。冻结检测 epsilon=0.1mm |
| **station_yard**（站院环境站） | device_status, data_gap, arrival_rate | **data_freeze**、**data_spike** | 站院监测温度/湿度/噪声，波动正常。主要做设备状态和缺失检测。站院设备本身的高温可触发 device_status |

### 4.3 同区域一致性比对规则

对于雨量站（rainfall）的 data_spike 场景，不直接产生突变告警，但引入**区域一致性比对**作为辅助检测：

```
条件：本站降水值 P_site，邻近站集合 {P1, P2, P3}
条件1：P_site > 20mm/h 且 所有 P_i < 2mm/h（i=1,2,3）
  → 生成 [可疑] 设备异常告警（device_status），等级 yellow
  → 工单内容："XX雨量站降水值明显偏离邻近站，建议检查传感器"
条件2：P_site = 0 且 所有 P_i > 10mm/h（i=1,2,3）
  → 生成 [可疑] 设备异常告警
  → 工单内容："XX雨量站降水为0但邻近站均有降雨，建议检查传感器（翻斗可能堵塞）"
```

## 5. 告警级别与工单自动流转规则

### 5.1 级别-流转映射

| 告警类型 | 默认级别 | 升级条件 | 工单流转类型 | 复核要求 |
|---------|---------|---------|------------|---------|
| data_gap | yellow | 超时>12周期→orange；>2h→red | **A级**（自动转工单） | 系统自动 |
| data_freeze | yellow | 同指标第3次触发→orange；3个指标同时冻结→red | **B级**（人工复核） | 值班员确认后再转工单 |
| data_spike | orange | z_score>10→red；多指标同时突变→red | **B级**（人工复核） | 值班员确认后再转工单 |
| device_status | yellow | 离线>4h→red | **A级**（自动转工单） | 系统自动 |
| arrival_rate | yellow | <60%→red | **A级**（自动转工单） | 系统自动 |

### 5.2 同站点合并规则

当同一站点产生多个告警时，合并策略（继承自当前代码逻辑并优化）：

1. **同metric合并**：同一测项未办结的告警只保留一条，`tracking_count` 累加
2. **不同metric合并**：同一站点的不同告警合并为一条告警消息，级别取最高级
3. **级别合并规则**：red > orange > yellow
4. **流转类型合并**：A级(自动) > B级(复核)——只要合并后的告警中包含 A 级类型，整条告警转为自动转工单
5. **告警风暴抑制**：同一站点30分钟内不生成超过3条独立告警，超出者合并

### 5.3 恢复检测

所有告警类型应支持**自动恢复检测**：

| 告警类型 | 恢复条件 | 自动恢复操作 |
|---------|---------|------------|
| data_gap | 收到新的有效数据包 | 告警状态→resolved |
| data_freeze | 最近6条数据 max-min > epsilon | 告警状态→resolved |
| data_spike | 后续5条数据回落到正常范围 | 告警状态→resolved（需人工确认） |
| device_status | 设备状态重新变为online 或 电压恢复正常 | 告警状态→resolved |
| arrival_rate | 次日到报率恢复到>90% | 告警状态→resolved（次日凌晨自动检查） |

## 6. 检测周期与去重策略

### 6.1 检测执行周期

| 检测类型 | 执行频率 | 说明 |
|---------|---------|------|
| data_gap | 每30秒 | 每次生成数据时实时判断 |
| data_freeze | 每30秒 | 每次生成数据时实时判断 |

| 去重维度 | 策略 | 说明 |

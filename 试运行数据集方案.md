## 二、数据关联规则（必须遵守的约束）

### 2.1 表间外键约束

```
sites.id ─┬── sensor_data.site_id
           ├── alerts.site_id
           ├── work_orders.site_id
           ├── inspection_plans.site_id
           ├── device_shadows.site_id
           ├── data_arrival.site_id
           ├── water_level_checks.site_id
           ├── maintenance_plans.site_id
           ├── inspection_schemes.site_id
           ├── timeline_events.site_id (? 部分有)
           └── user_sites.site_id

alerts.id ──── timeline_events.source_id (source_type='alert')
alerts.related_order_no ──── work_orders.order_no

work_orders.order_no ──── timeline_events.source_id (source_type='order')
                        ──── work_order_photos.order_no(? 如果存在)

inspection_plans.id ──── inspection_tasks.plan_id
```

### 2.2 计数一致性规则（最易出问题）

| 数据框 | 计数方式 | 需与谁一致 |
|-------|---------|-----------|
| 驾驶舱 `online_sites` | `SELECT COUNT(*) FROM sites WHERE status='online'` | 站点列表实际状态 |
| 驾驶舱 `active_alerts` | `SELECT COUNT(*) FROM alerts WHERE status='pending'` | 预警中心待处理数 |
| 驾驶舱 `open_orders` | `SELECT COUNT(*) FROM work_orders WHERE status NOT IN ('closed')` | 工单中心非关闭数 |
| 预警中心 `pending` | alert列表 status='pending' 的条数（全量筛选数据） | 驾驶舱 active_alerts |
| 预警中心 `today` | 当日创建的 alert 数 | 按 created_at 计算 |
| 工单中心 `total` | work_orders 总行数 | 工单列表 |
| 工单中心 `today` | 当日创建的工单数 | 按 created_at 计算 |
| 设备管理 | device_shadows 与 sites 的设备数 | 站点详情中的设备 |
| 站点详情 | `/api/site/status/<id>` 中的 devices | `/api/devices` 按site_id过滤 |
| 待复核 | alerts.flow_type='manual' AND flow_status='pending_review' | pending-review API |

### 2.3 驾驶舱站点卡片核心关联规则（最容易不自洽的部分）

驾驶舱的每个站点卡片，数据来源于 **4个独立查询的交叉**：

```
/data/realtime ─┬─ site_id ────── sites 表 (name, type, lat, lng)
                ├─ latest_value ── sensor_data 表 MAX(id) (value, metric, recorded_at)
                ├─ status ──────── device_shadows 表 LEFT JOIN (有offline设备=offline)
                └─ alert ───────── alerts 表 独立查询 ─→ alertSiteIds 映射
```

具体规则：
- 站点 status = offline ⇔ device_shadows 中该站至少1台设备 offline
- 站点是否出现在 issue 组 ⇔ (有pending告警 || status=offline || latest_value极端)
- 站点标签优先级: `离线 > 告警 > 数据缺失 > 数据异常 > (无标签)`
- `latest_value` 取值逻辑: `round(row['value'],2) if row else 0` — 无数据时=0 **不是null**
- `wl_value` 仅当 metric='water_level' 的 sensor_data 存在时出现

### 2.4 `/api/data/realtime` 响应字段映射

```python
# 每个站点对象包含（后端自动计算，无法手动定义响应）:
site_dict = {
    'id': s['id'],           # sites.id
    'code': s['code'],       # sites.code
    'name': s['name'],       # sites.name
    'type': s['type'],       # sites.type
    'lat': s['lat'],         # sites.lat
    'lng': s['lng'],         # sites.lng
    'status': 'online'|'offline',  # 来自 device_shadows 关联
    'latest_value': float,   # sensor_data MAX(id).value 或 0
    'latest_metric': str,    # sensor_data MAX(id).metric 或 ''
    'latest_unit': str,      # sensor_data MAX(id).unit 或 ''
    'latest_time': str,      # sensor_data MAX(id).recorded_at 或 ''
    # 如果该站有 water_level 数据，额外返回：
    'wl_value': float,       # sensor_data WHERE metric='water_level' MAX(id).value
    'wl_time': str,          # sensor_data WHERE metric='water_level' MAX(id).recorded_at
}
```

> ⚠️ 这意味着：不能直接定义 `/api/data/realtime` 的响应数据，而必须通过向 `sensor_data` 和 `device_shadows` 表插入正确数据来间接生成。

所有 `created_at` 字段必须是同一时间窗口内的（建议模拟最近 2-3 天的数据），以保证：
- "今日新增"计数正确
- 告警/工单排序正确
- 7天趋势图有数据可画
- "本月累计"有效

## 五、生成策略

### 5.1 实施方法
建议编写一个独立的 `seed_e2e.py` 脚本，按以下步骤执行：
1. **备份旧库**：先缩库再插入（或新建一个 `water-test.db`）
2. **清除种子检查**：跳过 `seed_data()/seed_alerts()` 等旧种子
3. **按表顺序插入**：sites → sensor_data → device_shadows → alerts → work_orders → inspections → inspection_tasks → data_arrival → timeline_events → weather → rainfall_forecast → water_level_checks → parts_inventory → maintenance_plans
4. **构建关联**：每张表的数据严格使用上述ID和关联键

### 5.2 数据量统计
- 站点: 6
- 传感器数据: 6（实时）+ 每个站点各追加20条历史（共120条）
- 设备: 11
- 告警: 5
- 工单: 3
- 巡检计划: 2 + 任务: 8
- 到报记录: 6
- 时间线: 15
- 天气: 24小时 + 降雨预报48条
- 水位校核: 2
- 备件库存: 5 + 日志若干
- **总计约 250条记录**（含历史传感器数据约120条）

### 5.3 关键脏数据注入点
为验证数据框联动，设以下触发条件：
- 告警中心 待处理数(pending=4) == 驾驶舱 active_alerts → 校验一致
- 告警中心 今日新增(today=4) == 当日创建告警数 → 校验一致
- 告警3(罗亭缺失)自动转工单后 → 工单列表出现新条目 → 关联查询验证
- 站点5(万家埠) offline → 驾驶舱online=5 → 站点列表标记红色 → 地图标记变色
